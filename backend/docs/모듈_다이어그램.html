2의존 방향 — "위에서 아래로만"

화살표는 한 방향뿐입니다. 코어가 동에 의존하면 순환이 생기고, 동끼리 직접 의존하면 충돌이 폭발합니다.

허용 — Client → FastAPI → 동 → 코어 서비스 → 외부/DB

금지 1   동 → 코어가 그 동을 알고 있음 (코어는 동을 모름)

금지 2   동 → 다른 동을 직접 import (옆 동의 service · models는 절대 import 안 함)

금지 3   동 → 다른 동의 DB 테이블을 직접 SELECT (UserContextService 통과 필수)

이 규칙 덕분에: ① 한 동을 빼도 시스템이 돌아감 (예: 토론 동 잠시 끄기 가능) ② 한 동만 따로 테스트 가능 ③ 6명이 동시에 PR 올려도 머지 충돌이 거의 없음

3동 간 통신 패턴 — 읽기 vs 쓰기

옆 동의 데이터가 필요할 때, 동기적으로 읽을 건지(관리사무소), 비동기적으로 쓰기 요청할 건지(쪽지함) 구분합니다.

①은 티어·관심사 같은 사용자 본질 상태. ②는 뉴스·시장지표 같은 다른 동의 도메인 콘텐츠. 어느 경우든 다른 동 테이블을 직접 SELECT하지 않습니다.

예시 ① ctx = await user_context.get_for_mentor_chat(user_id)
② news = await content_reader.get_today_news_for_user(user_id, top_k=3)

학습 동이 "사용자가 개념 퀴즈를 통과했다"는 사실을 성장 동에 알리고 싶을 때, 성장 동의 테이블을 직접 UPDATE하지 않고 이벤트를 발행합니다. 옆 동이 자기 테이블을 어떻게 바꿀지는 자기가 결정.

예시 학습 동: ConceptMasteredEvent 발행 →
성장 동이 구독: 자기 테이블의 진척도 컬럼을 직접 +1 업데이트
